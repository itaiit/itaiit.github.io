# 桥接模式

使用桥接模式的场景中：如果有两个独立变化的维度，可以在他们的抽象层面上建立关联，使他们各自独立变化而互不影响。

| 优势         | 说明                                                         |
| ------------ | ------------------------------------------------------------ |
| 避免类爆炸   | N 种抽象 + M 种实现 → 只需 N + M 个类，普通的需要 N * M 个类 |
| 扩展性强     | 新增颜色？只需加一个 `Color` 实现；新增形状？只需加一个 `Shape` 子类 |
| 符合开闭原则 | 对扩展开放，对修改关闭                                       |
| 解耦清晰     | 抽象和实现完全分离，各自独立演化                             |

**问题**：我们平时使用的在spring boot项目中，controller层依赖service的接口，这样可以有多个service实现，这是不是桥接模式？

这不是桥接模式，桥接模式的重点是抽象层面上建立关联，尽管我们将 controller 继承了一个抽象类，在抽象类中依赖 service 接口，这依然不能算是桥接模式。

桥接模式要求变化的维度是正交的，可以理解成是相互之间可以自由组合，依赖性很小。而对于 controller 和 service 来说，我们的一个 controller 往往强耦合一个固定的 service。而不会自由组合其他的 service。
